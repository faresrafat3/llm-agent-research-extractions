# Tree of Thoughts — Research / System Summary

Paper: https://arxiv.org/abs/2305.10601  
Venue: NeurIPS 2023  
Code: https://github.com/princeton-nlp/tree-of-thought-llm  
Audited commit: `8050e67d0e3a0fddc424d7fa5801538722a4c4cc`

## Core idea

Language models usually decide token-by-token left-to-right. That fails on problems that need **exploration, lookahead, or backtracking**.

**Tree of Thoughts (ToT)** reframes LM inference as search over a tree where:

- each **node** is a partial solution state (a sequence of “thoughts”),
- each **edge** is a thought generated by the LM,
- a **state evaluator** (also the LM) scores how promising a node is,
- a **search algorithm** (BFS beam or DFS with prune) expands the tree and keeps the best frontier.

ToT generalizes CoT (single path of thoughts) and self-consistency (sample many full solutions and vote at the end) by evaluating *partial* solutions and pruning early.

## Paper algorithm (conceptual)

Inputs: problem `x`, thought generator `G`, state evaluator `V`, search algorithm.

1. Maintain a frontier of states (initially empty thought on `x`).
2. For each step / expansion:
   - **Generate** candidate next thoughts for each frontier state (`sample` or `propose`).
   - **Evaluate** new states (`value` independently or `vote` comparatively).
   - **Select** top-`b` states (greedy or sample by value).
3. Return best complete trajectory(ies).

## Key components

| Component | Role |
|---|---|
| `solve` | ToT+BFS main loop: generate → evaluate → select for `task.steps` |
| `naive_solve` | IO or CoT baseline sampling without tree search |
| `get_proposals` | Sequential next-step generation (Game24 / Crosswords) |
| `get_samples` | Independent full/partial samples (Creative Writing) |
| `get_values` | Per-state value with cache + duplicate-zeroing |
| `get_votes` | Comparative multi-candidate voting |
| `Game24Task` | 4-number puzzles, sympy check for 24 |
| `TextTask` | Forced end-sentence paragraphs + coherency scoring |
| `MiniCrosswordsEnv/Task` | Board env, propose+value, DFS in notebook |

## Generation modes

- **`sample`**: draw `n_generate_sample` independent completions (standard or CoT prompt). Used in creative writing.
- **`propose`**: one call lists several possible *next* steps; parsed into children. Used in Game of 24 and crosswords.

## Evaluation modes

- **`value`**: score each candidate alone → map labels to numbers  
  Game24: `impossible=0.001`, `likely=1`, `sure=20`  
  Crosswords: sure/maybe/impossible counts for pruning.
- **`vote`**: one prompt shows all candidates; model picks best choice id; aggregate votes.

## Selection modes

- **`greedy`**: keep top `n_select_sample` by value (paper beam width `b`).
- **`sample`**: multinomial sample proportional to values.

## Headline paper results (context)

- Game of 24: GPT-4 CoT ~4% → ToT ~74% (repro log later ~69% due to sampling).
- Creative writing and mini crosswords also improve via deliberate search.

## Strengths for extraction / ARSENAL

- Extremely clean prompt surface (three prompt modules + one BFS file).
- Clear decision taxonomy: generate × evaluate × select.
- Direct ancestor of LATS (add env reward, UCT, reflection).
- Fits ARSENAL L3 (tree search) as the classic non-MCTS baseline.

## Inputs / Outputs

**Inputs:** task idx / problem string, backend model, temperature, method flags, sample counts.  
**Outputs:** candidate trajectories `ys`, step-wise infos, reward via `test_output`, token cost via `gpt_usage`.
